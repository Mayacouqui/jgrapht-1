JGraphT Project Policies


The intent of these policies is to direct the JGraphT development in an orderly fashion. These are by no means "hard-rules" and they can be changed. If you have suggestions of how to do things better, please send them over. However, until any such suggestions are accepted you are expected to conform to these policies.

1.  Credits & Ego

1.1.  You must retain the original authorship of files even if you rewrote them ground-up and did not use any of the original. You probably built on mistakes of the original author/s and they still deserve the credits for their pioneering work.

1.2.  Fixing bugs in someone else's code or adding little features does not entitle you co-authorship. However, if you made substantial contribution you may add your name at the end of the author list.

1.3.  Expect your code to change and evolve. None of us do everything right the first time even if we're great programmers, which of course we are :). Please welcome the changes others introduce to your code and try to judge them in an "ego-neutral" manner.

2.  Coding Conventions

2.1.  "One consistent look". The code of JGraphT should have one consistent look. It is important since JGraphT is a class library, and not a closed product. Our users will look into the code very often. If we confuse them with many coding styles they will hate us and underestimate our professionalism. If we give them one consistent look, they will love us (even if our styling will differ from their taste). As a general rule, watch existing classes and do alike.

2.2.  Automated tools. The "Jalopy" and "Checkstyle" tools are used and greatly help to achieve one consistent look -- it is strongly recommended that you use them. The both have plugins for various IDEs. You can find more about them in the "etc" folder of JGraphT.

2.3.  "Principle of least surprise" should be practiced. A method should do what its name suggests and should not have unrelated side effects. Try to make things expected as an alternative to referring users to the documentation. Ideally, users should be able to do an "auto-complete" programming with JGraphT. Ideally, they would not need to read the documentation.

2.4.  "Simplicity is gold". Your code should be as simple as possible (especially the public API for our users). "Simple" does not mean stupid. If you need to do clever things, do clever things. However, don't do clever if you can do simple!

2.5   Professional finishing. Your release-ready code should not raise compiler warnings (to the extent possible). It should include a complete Javadoc in a way that execution of  'ant javadoc'  will output no complains concerning it. If the code has external influences (e.g., additional libraries, variations to ANT script) please make sure they are taken care of.

3.  Member's dos and don'ts

3.1.  JGraphT is a nice project of nice people. It is a strictly no-flame project! A flame incident, whether against other members or against JGraphT users, will result in an immediate removal of the involved member from the JGraphT team. The authorship credits of such member might be renamed to "unascribed".

3.2.  Members should not modify the JGraphT project website - this task is done by the project admin.

3.3.  New members should NOT commit changes to CVS before reviewed by other members. This is done in order to allow new members to receive feedback on their code before committing, hence prevent creating CVS mess on "startup mistakes".

3.4  Member should not commit to CVS a code that does not compile. The code should be compatible with JDK1.3 or above.

3.5.  Lastly, the members in the JGraphT project may be of different levels of experience and proficiency. If you are a "wizard", be kind to and patient with the "newbies" - you've also been a newbie once. If you are a less experienced developer, learn to appreciate the inputs from the "wizards" - these can be important and valuable contributions for your own professional growth.


That's all there is to it!
