JGraphT Development Plan


This is a working document where the development plan is maintained. The document is sorted according to versions in descending order (newest version at the top). For each version there is a list of development items related to that version. These items have one of the following statuses* attached to them:

proposed A proposed development item for a release. This status mark includes initials of the person who proposed it. After mail discussion, such item will either be committed, rejected, or deferred.
committed A committed item is one that we have decided to address for a release.
deferred A reasonable proposal that did not make it in to this release for some reason. It is marked as deferred with a brief note as to why they were deferred. Deferred plan items may resurface as proposed plan items at a later point.
rejected Plan items that were proposed but judged unworkable are marked as rejected plan items, with an accompanying summary of why they were dismissed. Keeping track of rejected items avoids repeating the discussion.
* these statuses were adopted from the Eclipse project plan.

Please try to use simple and tidy HTML for this document, to ease version compare.

 

Following Versions

Automatic graph layout when visualizing via the JGraph adapter. Employ the ready-made JGraph layouts to a achieve a quick and automatic way to take a given graph and visualize it. (proposed BN)

Integration with GraphViz

GraphViz is a nice tool for automated layout. Discussion: The Java port (grappa)
looks pretty useless since it doesn't even do layout, but another
possible integration method is to generate files in their format (dot)
and exec GraphViz . For a live display rather than a static image,
you'd have to tell GraphViz to produce dot output, and then parse it for
the location info. (proposed JVS)
As a first lame integration, it's enough that we generate the dot files and automatically post them against a GraphViz server (like the demo at http://odur.let.rug.nl/~vannoord/fsademo/). This will keep us platform independent and will allow users to see their graphs (however, only in browser). (proposed BN)

 

Version 0.4.2

Graph trigger listeners and graph assertions. Discussion: GraphTriggerListener extends GraphListener, receives "before" events, and can throw (unchecked) exceptions to abort the action. Graph assertions are implementations of GraphTriggerListener which enforce graph properties. an AbstractGraphAssertion class will isolate the exception handling from the logical checks, and make it easy to use composites. Methods of that class will redirect to a testing method that will return boolean flag to indicate an approved operation. Example: the AbstractGraphAssertion receives the 'beforeVertexAdded(v)' call, and tests the approveAddVertex(v) method - if true it returns, if false throws some assertion exception. The approveAddVertex(v) is, of course, polymorphic. (proposed JVS+BN)
